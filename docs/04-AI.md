# 04 — AI & Extraction Architecture

### Status: **ARCHITECTURE FROZEN**

## 1. Provider & Model Selection

* **Primary Provider**: Google Gemini API (`@google/generative-ai`)
* **Primary Model**: `gemini-2.5-flash` (or `gemini-1.5-flash` depending on API region availability)
* **Rationale**:
  * **Cost-Efficiency**: Generous free-tier limits (up to 15 RPM / 1 million tokens per minute) make it ideal for free deployment.
  * **Speed**: Latency is extremely low (averaging 1-2s for batch mapping).
  * **Native Structured Outputs**: Built-in JSON-mode support guarantees that the model returns valid JSON conforming to our target schema.

---

## 2. Structured Output Strategy

We will configure the Gemini model client with `responseMimeType: "application/json"` and supply a strict JSON schema template to enforce shape.

```typescript
const responseSchema = {
  type: "object",
  properties: {
    records: {
      type: "array",
      items: {
        type: "object",
        properties: {
          created_at: { type: "string" },
          name: { type: "string" },
          email: { type: "string" },
          country_code: { type: "string" },
          mobile_without_country_code: { type: "string" },
          company: { type: "string" },
          city: { type: "string" },
          state: { type: "string" },
          country: { type: "string" },
          lead_owner: { type: "string" },
          crm_status: { 
            type: "string", 
            enum: ["GOOD_LEAD_FOLLOW_UP", "DID_NOT_CONNECT", "BAD_LEAD", "SALE_DONE", ""]
          },
          crm_note: { type: "string" },
          data_source: { 
            type: "string", 
            enum: ["leads_on_demand", "meridian_tower", "eden_park", "varah_swamy", "sarjapur_plots", ""]
          },
          possession_time: { type: "string" },
          description: { type: "string" }
        },
        required: [
          "created_at", "name", "email", "country_code", "mobile_without_country_code",
          "company", "city", "state", "country", "lead_owner", "crm_status", 
          "crm_note", "data_source", "possession_time", "description"
        ]
      }
    }
  },
  required: ["records"]
};
```

If the LLM returns an empty string or fields that don't match the enum parameters, they will be coerced or sanitized during backend **Zod validation** post-extraction.

## 3. System Prompt Instructions

The template inside `server/src/features/importer/extraction.prompt.ts` will direct the model with the following rules:

1. **Role**: Act as a CRM data ingestion specialist for GrowEasy CRM.
2. **Column Matching**: Map incoming fields to targets using semantic synonyms:
   * `name` <= "Full Name", "Customer", "Contact Person", "Client Name"
   * `email` <= "E-mail Address", "Mail Id", "Primary Email"
   * `mobile_without_country_code` <= "Phone Number", "Mobile", "Contact", "Telephone"
3. **Phone Splitting**:
   * Inspect phone strings (e.g. "+91 9876543210", "+1 (555) 019-2834").
   * Separate country codes (e.g. `+91`, `+1`) into the `country_code` field.
   * Strip all spaces, parentheses, dashes from the subscriber number, and save it in `mobile_without_country_code`.
4. **CRM Status Mapping**:
   * Map raw status values to: `GOOD_LEAD_FOLLOW_UP`, `DID_NOT_CONNECT`, `BAD_LEAD`, or `SALE_DONE`.
   * If a value cannot be matched confidently, default to `""` (will be coerced to `null` in validation).
5. **Data Source Mapping**:
   * Map sources to: `leads_on_demand`, `meridian_tower`, `eden_park`, `varah_swamy`, or `sarjapur_plots`.
   * If a value cannot be matched confidently, set to `""`.
6. **Date Standardizing**:
   * Map date fields to `created_at`. Normalize them so that they are parseable via JavaScript (`new Date(val)` does not return `Invalid Date`).
7. **Multiple Contacts Handling**:
   * If multiple email addresses are present: use the first one for `email` and append the rest to `crm_note` as `[Emails: secondary@example.com]`.
   * If multiple phone numbers are present: use the first one for the phone number and append the rest to `crm_note` as `[Phones: +919876543212]`.
8. **Note Ingestion**:
   * Pack any additional remarks, custom responses, or unmapped custom properties into the `crm_note` field so that no user data is lost.
9. **Row Skipping Guard**:
   * **CRITICAL**: If a row contains **neither** an email address nor a phone number, it must be excluded from the `records` list.

## 4. Backend Programmatic Verification

To prevent hallucinations, the backend `importer.controller.ts` will run a secondary programmatic filter on the LLM's output. Any record that has:
* `email` empty OR missing `z.string().email()` format **AND**
* `mobile_without_country_code` empty OR less than 5 characters long
will be filtered out and moved to the "skipped" list. This double-layered safety guard ensures zero invalid contact entries enter the final UI results.
